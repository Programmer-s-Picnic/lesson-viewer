Complete explanation for beginners

Read this section if you want to understand what the project does and how it works.

Beginner friendly

What this project is

This project is a browser-based face recognition system. That means it works inside web pages using the camera and JavaScript. It does not start with automatic knowledge of who a person is. First, you must give it sample face images for each person. After that, the system tries to compare the live camera face with the saved examples and decide which known person it matches best.

In simple words, the project has two main jobs. The first job is to collect example faces. The second job is to recognize faces in the camera using those examples. That is why this hub links to two different pages: one for capture and one for recognition.

Why there are separate pages

Keeping capture and recognition separate makes the system easier to understand, easier to debug, and safer to use. The capture page is only for preparing training data. The recognition page is only for live use. When these responsibilities are separated, it becomes much easier to check where a problem is happening.

For example, if recognition is not working, you can ask simple questions: Did I save enough images? Did I export the JSON file? Did I place the images in the correct folder? Did the recognition page load the files properly? This separation reduces confusion.

What happens in capture.html

The training-data collection page.

You enter a person name such as Champak.
The camera starts and the page detects a face box.
When you press Save Face, the system crops the face region and stores it locally in the browser.
Each saved image also gets a file name so that it can later be placed in the correct folder.
Finally, the page can export a grouped JSON file that lists each person and the image files connected to that person.

What happens in recognize.html

The live recognition page.

The page loads the face detection and face recognition models.
It reads data/faces.json to know which people and image files should be used for training.
It loads the listed images from images/faces/ and tries to extract face descriptors from them.
If training succeeds, it builds a matcher so that live faces can be compared against the known people.
Then it starts the camera, draws rectangles around faces, and attempts to show names for recognized people.

Understanding the files in simple words

These are the main files and folders you need.

index.html is the main hub page. It explains the system and gives links to the other tools.
capture.html is the page where you collect example face images for each person.
recognize.html is the page where you start the camera and try live recognition.
data/faces.json is the training data map. It tells the recognition page which people exist and which image files belong to them.
images/faces/... is the folder where the actual face image files must be stored.

What is faces.json

This file is very important because it connects names to images.

Key file

The recognition page needs a simple way to know which images belong to which person. That is the job of faces.json. It is a structured data file. Each entry usually contains a person name and a list of image files for that person.

A typical item looks like this: { name: "Champak", files: ["images/faces/champak-1.jpg", "images/faces/champak-2.jpg"] }

When the recognition page opens, it reads this file, loads those images, tries to find faces in them, and uses them to build the training set. If the file is missing, badly written, or points to the wrong image paths, recognition will not work correctly.

Common checks

Things to verify when labels do not appear.

Make sure camera access is allowed and the page is served over a secure context.
Check that data/faces.json exists and that its image paths match the actual files.
Use 3 to 5 clear face images per person with decent lighting and visible facial features.

Common problems and what they mean

These explanations help non-technical users understand errors.

Troubleshooting

Why do I see rectangles but no names?

This usually means face detection is working, but recognition is not ready. The training images may not have loaded, the JSON file may be missing, the file paths may be wrong, or the training images may not contain clear faces.

Why does the page say Unknown?

Unknown means the system could not confidently match the live face with the saved training descriptors. This can happen when training images are too few, too blurry, badly lit, or when the person in the camera looks very different from the saved examples.

Why does camera access fail?

The browser may not have permission, the device may not have an available camera, or the page may not be running in a secure environment. Camera-based pages work best on HTTPS or supported localhost setups.

Why are good training images important?

The model learns from the sample images you provide. If those images are dark, cut off, blurry, or inconsistent, the system cannot build reliable descriptors. Clear front-facing images with decent lighting produce much better results.

Folder expectations

The recognition page expects these paths.

index.html → this hub page
capture.html → training image capture page
recognize.html → live recognition page
data/faces.json → grouped training metadata
images/faces/... → saved face crops referenced by the JSON file
The paths matter. If the JSON points to files that do not exist at the expected location, the recognition page cannot train properly. A large part of successful setup is simply making sure the folder structure is correct.

Best practices for better results

Simple habits that improve recognition quality.

Recommended
Capture multiple images per person instead of depending on only one.
Use clear lighting and avoid very dark or very bright backgrounds.
Keep the face visible and avoid heavy cropping of the forehead or chin.
Try slight variation in angle and expression, but keep the face easy to see.
Rebuild training whenever you change the JSON file or replace images.

Final understanding

The simplest way to remember the full system.

Think of the project like this: first you teach, then you test. capture.html is the teaching part because you give the system sample faces. recognize.html is the testing part because the system looks at the live camera and tries to decide which known person it is seeing.

This hub page exists so that anyone opening the project can quickly understand the purpose of each page, the role of each file, the correct order of use, and the most common reasons why recognition may fail. That makes the whole project easier to use, easier to explain, and easier to maintain.
