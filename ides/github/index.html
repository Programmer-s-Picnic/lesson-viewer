Getting started

GitHub is a platform for hosting Git repositories and collaborating on code. You‚Äôll typically use Git locally and GitHub as the remote (online) home for your project.

1) Create your account + set up your profile
  • Add a profile photo and bio (helps collaborators recognize you).
  • Set your display name and public email preferences.
  • Add a pinned repository (showcase your best project).
2) Secure your account
  • Enable Two-Factor Authentication (2FA).
  • Use a password manager.
  • Prefer SSH keys or fine-grained tokens over passwords for Git operations.

Install Git (local)

You need Git on your computer to commit and push code. After installing, configure your name + email: 

git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main
git config --global pull.rebase false
Tip: If you contribute to open source, consider using a ‚Äúno-reply‚Äù GitHub email in your Git config to protect privacy.

SSH keys (recommended)

SSH avoids entering passwords and is widely used for GitHub auth.

# Generate a key (ed25519 is modern/recommended)
ssh-keygen -t ed25519 -C "you@example.com"

# Start agent and add key
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

# Copy public key and add it to GitHub: Settings ‚Üí SSH and GPG keys
cat ~/.ssh/id_ed25519.pub

Repositories

A repository (repo) contains your project‚Äôs code, history, and collaboration features (issues, pull requests, actions).

Create a repository

On GitHub (web)
  • New repository ‚Üí choose name + visibility (Public/Private).
  • Add README (project intro), LICENSE, and .gitignore.
  • Use a clear description and topics (tags) for discoverability.
Locally (Git-first)
  • Create a folder, initialize Git, commit, then add GitHub as remote.
  • Best when you already have a project and want to publish it.

Clone vs Fork

Clone

Download a repo you have access to (yours or someone else‚Äôs) onto your computer.

git clone git@github.com:OWNER/REPO.git
Fork

Create your own copy of someone else‚Äôs repo under your account (common in open source).

# After forking on GitHub, clone your fork:
git clone git@github.com:YOURNAME/REPO.git

Repository essentials

  • README.md: What it is, how to run it, examples, screenshots, links.
  • LICENSE: Legal terms for use/contribution (MIT/Apache-2.0/GPL, etc.).
  • .gitignore: Files you don‚Äôt want tracked (node_modules, build, secrets).
  • CONTRIBUTING.md: How others can contribute (setup, style, PR process).
  • CODEOWNERS: Auto-requests reviews from specific people/teams.

Git basics you‚Äôll use daily

Core flow 
git status
git add .
git commit -m "Describe the change"
git push

Repeat this loop as you work. Commit messages should explain why and what.

Pull latest changes 
git pull

If you collaborate with others, pull often. If you prefer a linear history, you can use git pull --rebase.

Remotes (origin)

git remote -v
git remote add origin git@github.com:OWNER/REPO.git
git push -u origin main

Viewing history

git log --oneline --decorate --graph --all
Rule of thumb: Never commit secrets (API keys, passwords). Use environment variables and secret managers.

Branching (how teams avoid chaos)

A branch is an independent line of development. Most teams keep main stable and do work on feature branches.

Typical feature-branch workflow

# Start from main
git switch main
git pull

# Create a new branch
git switch -c feature/login-form

# Work, commit, push
git add .
git commit -m "Add login form UI"
git push -u origin feature/login-form

Merging vs Rebasing (what to choose)

Merge (safe + common)

Keeps branch history, creates a merge commit (or fast-forward).

git switch main
git pull
git merge feature/login-form
Rebase (clean history)

Replays your commits on top of main. Great locally; be careful rebasing shared branches.

git switch feature/login-form
git fetch origin
git rebase origin/main
Important: Avoid rebasing branches that other people already pulled, unless your team explicitly agrees on it.

Pull requests (PRs)

A pull request is a proposal to merge code into another branch (usually main). PRs enable review, discussion, and automated checks before merging.

Making a great PR

Title + description
  • Write a clear title: ‚ÄúFix login validation‚Äù not ‚ÄúUpdate‚Äù.
  • Explain the problem, solution, and any trade-offs.
  • Link related issues (e.g., ‚ÄúCloses #123‚Äù).
Keep it reviewable
  • Smaller PRs merge faster and reduce bugs.
  • Add screenshots or GIFs for UI changes.
  • Include tests or explain why not needed.

PR merge strategies

  • Merge commit: preserves full branch history.
  • Squash and merge: turns many commits into one (clean main history).
  • Rebase and merge: linear history, keeps individual commits.

Reviews & checks

Use review comments to ask questions, suggest changes, and ensure style consistency. Use CI checks (tests/lint) to prevent regressions.

Issues & Projects (planning + tracking)

Issues capture tasks, bugs, enhancements, and discussions. Projects helps organize issues/PRs in a board (Kanban-style).

Issue best practices

Use templates
  • Bug report: steps to reproduce, expected vs actual, environment.
  • Feature request: problem, proposed solution, alternatives.
Use labels + milestones
  • Labels: bug, enhancement, good first issue
  • Milestones: group issues for a release (v1.2, sprint).

Link issues and PRs

# In PR description:
Closes #42
Fixes #105
Resolves OWNER/REPO#12
Tip: ‚ÄúGood first issue‚Äù + clear reproduction steps is one of the fastest ways to get helpful open-source contributions.

Collaboration: permissions & teamwork

GitHub lets you control who can read/write/admin a repository. In organizations, you typically manage access using teams.

Common permission model

  • Read: view and clone.
  • Triage: manage issues and PRs (no write).
  • Write: push to repo (often protected by branch rules).
  • Maintain: manage repo settings (limited admin).
  • Admin: full control.

Branch protection rules (highly recommended)

Protect main
  • Require pull requests before merging.
  • Require review approvals.
  • Require status checks (CI tests).
Quality gates
  • Require linear history (optional).
  • Require signed commits (optional).
  • Require conversation resolution.

GitHub Actions (CI/CD automation)

Actions run workflows (defined in YAML) when events happen: pushes, PRs, schedules, releases. Use them to run tests, lint, build, and deploy automatically.

Basic workflow example (Node.js)

# .github/workflows/ci.yml
name: CI
on:
  pull_request:
  push:
    branches: [ "main" ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - run: npm ci
      - run: npm test

Key concepts

  • Workflow: the YAML file.
  • Job: a set of steps that runs on a runner.
  • Step: a command or an action.
  • Runner: where it runs (GitHub-hosted or self-hosted).
  • Secrets: encrypted variables used in workflows (API keys, tokens).
Security: Be careful with pull requests from forks‚Äîdon‚Äôt expose secrets to untrusted code.

GitHub Pages (hosting)

GitHub Pages hosts static sites directly from a repository. Great for docs, portfolios, and demos.

Quick setup

  1. Repo ‚Üí Settings ‚Üí Pages
  2. Select source: Deploy from a branch (commonly main / docs/) or GitHub Actions
  3. Pick folder (root or /docs) and save

Common patterns

Docs site
  • Put documentation in /docs or use a docs generator.
  • Use Actions to build and publish if needed.
Frontend build
  • Build React/Vue/Angular and deploy the build output.
  • Actions can upload the build artifact to Pages.

Releases & tags

Releases help you publish stable versions of your project. A tag marks a specific commit (e.g., v1.0.0).

Semantic Versioning (SemVer)

MAJOR breaking changes ‚Ä¢ MINOR new features, backwards compatible ‚Ä¢ PATCH bug fixes

Create a tag + push it

git tag -a v1.0.0 -m "First stable release"
git push origin v1.0.0

Generate a CHANGELOG

Keep a CHANGELOG.md so users can quickly see what changed between versions.

Security essentials

GitHub has built-in features to reduce risk: dependency alerts, secret scanning, code scanning, and security policies.

Do these in every repo

Protect secrets
  • Never commit secrets; rotate immediately if you do.
  • Use repository/organization secrets for Actions.
  • Use .env locally and add it to .gitignore.
Dependency hygiene
  • Enable Dependabot updates (where available).
  • Review security alerts and patch quickly.
  • Pin critical dependencies if needed.

Security policy + reporting

Add SECURITY.md explaining how to report vulnerabilities. This builds trust with users.

Pro tip: Use least-privilege permissions for tokens and Actions workflows.

Advanced Git (super useful in real life)

Stash work-in-progress

git stash
git pull
git stash pop

Undo changes safely

Undo a commit (keep history)

Creates a new commit that reverses another commit (safe for shared branches).

git revert <commit_sha>
Reset (be careful)

Moves branch pointer; can rewrite history. Avoid on shared branches.

git reset --soft HEAD~1   # keep changes staged
git reset --mixed HEAD~1  # keep changes unstaged
git reset --hard HEAD~1   # discard changes

Recover ‚Äúlost‚Äù commits

git reflog
# find the commit, then:
git checkout <sha>
# or move branch back:
git reset --hard <sha>

Cherry-pick a commit

git cherry-pick <commit_sha>

Large files

Consider Git LFS for large binaries (videos, datasets). Don‚Äôt bloat your repo with huge files unless needed.

Troubleshooting (common problems)

Merge conflicts

  1. Open the conflicted files; look for <<<<<<< and >>>>>>>.
  2. Edit to keep the correct final content.
  3. git add <file> then git commit (or continue rebase).

‚ÄúPermission denied (publickey)‚Äù

# Check SSH connectivity
ssh -T git@github.com

# Ensure correct key is loaded
ssh-add -l

# Use the SSH remote URL
git remote set-url origin git@github.com:OWNER/REPO.git

Detached HEAD

You checked out a commit, not a branch. Create a branch from there:

git switch -c fix/detached-head

Accidentally committed secrets

Do this immediately: rotate the secret (invalidate old key), remove it from the repo, and consider rewriting history if necessary. If the repo is public, assume the secret is compromised.

Glossary

  • Remote: A hosted copy of your repo (e.g., GitHub).
  • Origin: Default name for your remote.
  • Upstream: The original repo you forked from (common in open source).
  • Commit: A snapshot of changes with a message + ID (SHA).
  • Branch: A pointer to a series of commits.
  • PR (Pull Request): Proposal to merge changes into a target branch.
  • CI/CD: Automated testing/building/deploying.
  • Tag: Named pointer to a commit (often a version).
