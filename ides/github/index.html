Getting started

GitHub is a platform for hosting Git repositories and collaborating on code. You‚Äôll typically use Git locally and GitHub as the remote (online) home for your project.

1) Create your account + set up your profile
  • Add a profile photo and bio (helps collaborators recognize you).
  • Set your display name and public email preferences.
  • Add a pinned repository (showcase your best project).
2) Secure your account
  • Enable Two-Factor Authentication (2FA).
  • Use a password manager.
  • Prefer SSH keys or fine-grained tokens over passwords for Git operations.

Install Git (local)

After installing, configure your name + email:

git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main
git config --global pull.rebase false
Tip: If you contribute to open source, consider using a ‚Äúno-reply‚Äù GitHub email in your Git config to protect privacy.

SSH keys (recommended)

SSH avoids entering passwords and is widely used for GitHub auth.

# Generate a key (ed25519 is modern/recommended)
ssh-keygen -t ed25519 -C "you@example.com"

# Start agent and add key
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

# Copy public key and add it to GitHub: Settings ‚Üí SSH and GPG keys
cat ~/.ssh/id_ed25519.pub

Repositories

A repository (repo) contains your project‚Äôs code, history, and collaboration features (issues, pull requests, actions).

Create a repository

On GitHub (web)
  • New repository ‚Üí choose name + visibility (Public/Private).
  • Add README, LICENSE, and .gitignore.
  • Use a clear description and topics for discoverability.
Locally (Git-first)
  • Initialize Git locally, commit, then add GitHub as remote.
  • Best when you already have a project and want to publish it.

Clone vs Fork

Clone

Download a repo you have access to (yours or someone else‚Äôs) onto your computer.

git clone git@github.com:OWNER/REPO.git
Fork

Create your own copy of someone else‚Äôs repo under your account (common in open source).

# After forking on GitHub, clone your fork:
git clone git@github.com:YOURNAME/REPO.git

Git basics you‚Äôll use daily

Core flow 
git status
git add .
git commit -m "Describe the change"
git push

Commit messages should explain why and what.

Pull latest changes 
git pull

If you prefer a linear history, you can use git pull --rebase.

Viewing history

git log --oneline --decorate --graph --all
Rule of thumb: Never commit secrets (API keys, passwords). Use environment variables and secret managers.

GitHub Desktop (GUI workflow)

GitHub Desktop is a free app (Windows/macOS) that lets you use Git and GitHub with a visual interface. It‚Äôs great for beginners, and also useful for experienced users when you want a fast, safe UI for common tasks.

What GitHub Desktop is best for

Everyday work
  • Clone repositories with one click.
  • See changed files clearly before committing.
  • Commit, push, pull, and fetch easily.
Branching & PRs
  • Create/switch branches visually.
  • Publish a branch to GitHub.
  • Create a Pull Request from the app.

Common Desktop flow (step-by-step)

  1. Sign in to GitHub inside GitHub Desktop.
  2. Clone a repo (File ‚Üí Clone Repository) or create a new one.
  3. Make changes in your editor.
  4. In Desktop, review changes ‚Üí write a summary + optional description ‚Üí Commit to branch.
  5. Click Push origin (or Publish branch the first time).
  6. Click Create Pull Request to open a PR on GitHub.

Fetch vs Pull (in Desktop)

Fetch origin

Checks for updates on the remote (safe). Doesn‚Äôt change your files.

Pull origin

Downloads updates and merges/rebases them into your current branch.

Handling conflicts

If a pull/merge causes conflicts, Desktop will show which files are conflicted. Open them in your editor, resolve, then mark them as resolved and continue the merge.

Tip: Desktop integrates nicely with editors (VS Code, etc.). Set your preferred editor in Desktop settings.

Branching (how teams avoid chaos)

A branch is an independent line of development. Most teams keep main stable and do work on feature branches.

Typical feature-branch workflow

# Start from main
git switch main
git pull

# Create a new branch
git switch -c feature/login-form

# Work, commit, push
git add .
git commit -m "Add login form UI"
git push -u origin feature/login-form

Merging vs Rebasing (what to choose)

Merge (safe + common) 
git switch main
git pull
git merge feature/login-form
Rebase (clean history) 
git switch feature/login-form
git fetch origin
git rebase origin/main
Important: Avoid rebasing branches that other people already pulled, unless your team explicitly agrees on it.

Pull requests (PRs)

A pull request is a proposal to merge code into another branch (usually main). PRs enable review, discussion, and automated checks before merging.

PR merge strategies

  • Merge commit: preserves full branch history.
  • Squash and merge: turns many commits into one (clean main history).
  • Rebase and merge: linear history, keeps individual commits.

GitHub Actions (CI/CD automation)

Actions run workflows (defined in YAML) when events happen: pushes, PRs, schedules, releases.

Basic workflow example (Node.js)

# .github/workflows/ci.yml
name: CI
on:
  pull_request:
  push:
    branches: [ "main" ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - run: npm ci
      - run: npm test

Glossary

  • Remote: A hosted copy of your repo.
  • Origin: Default name for your remote.
  • Upstream: Original repo you forked from.
  • Commit: A snapshot of changes.
  • Branch: A pointer to a series of commits.
  • PR: Proposal to merge changes.
