Basics & layout

VS Code is a fast editor with IDE-like features. Most power comes from Command Palette, extensions, and consistent workflows.

Key areas

  • Activity Bar: Explorer, Search, Source Control, Run/Debug, Extensions.
  • Side Bar: file tree, git changes, debug panels, etc.
  • Editor: tabs, splits, breadcrumbs, minimap.
  • Panel: Terminal, Output, Problems, Debug Console.
  • Status Bar: language mode, line endings, indentation, git branch, errors/warnings.
üí° Tip
Learn one command: Ctrl+Shift+P (or ‚åò+Shift+P) to open the Command Palette. It‚Äôs the fastest way to discover features.

Command Palette

  • Run commands by name: ‚ÄúFormat Document‚Äù, ‚ÄúToggle Word Wrap‚Äù, ‚ÄúGit: Clone‚Äù, ‚ÄúPreferences: Open Settings (JSON)‚Äù.
  • Prefix with > to run commands (it appears automatically when opened).
  • Use @ to jump to symbols (functions/classes) in the current file.
  • Use : to jump to a line number (or use Go to Line shortcut).

Settings & profiles

  • Settings UI is easiest for browsing options.
  • settings.json is best for portability and sharing.
  • Profiles let you switch sets of settings/extensions (e.g., ‚ÄúWeb‚Äù, ‚ÄúPython‚Äù, ‚ÄúMinimal‚Äù). 
{
  "editor.formatOnSave": true,
  "editor.tabSize": 2,
  "editor.rulers": [80, 100],
  "files.trimTrailingWhitespace": true,
  "files.insertFinalNewline": true
}

Workspaces

  • A folder is the simplest project type in VS Code.
  • A workspace (.code-workspace) can contain multiple folders + workspace-specific settings.
  • Keep project-specific settings in .vscode/settings.json (inside the repo) when helpful.
‚ÑπÔ∏è Note
Workspace settings override user settings, which is useful for standardizing formatting and tooling across a team.

Editing power features

Fast editing essentials

  • Multi-cursor to edit many places at once.
  • Rename symbol to refactor safely.
  • Go to Definition and Peek to understand code quickly.
  • Emmet for HTML/CSS (type shorthand, expand to full markup).
  • IntelliSense (autocomplete + docs).

Quality-of-life toggles

  • Word Wrap for long lines.
  • Render Whitespace to spot trailing spaces.
  • Format Document to keep code style consistent.
  • Problems panel for errors/warnings across files.

Multi-cursor & selection tricks

  • Add cursor: hold Alt (Option on macOS) and click.
  • Add cursor above/below: extremely useful for column edits.
  • Select next occurrence: select a word, then keep adding matches.
  • Column selection: hold Shift+Alt and drag (Windows/Linux), or Shift+Option (macOS).
üí° Tip
A common workflow: select a variable name ‚Üí Select Next Occurrence a few times ‚Üí type once to change multiple spots. For safer renames across a project, prefer Rename Symbol.

Refactor & navigation

  • Go to Definition / Peek Definition to inspect without leaving your file.
  • Go to References to see all usages.
  • Rename Symbol to refactor identifiers safely (needs language support/extension).
  • Code Actions (quick fixes) for imports, refactors, lint fixes.
  • Breadcrumbs show file path + symbol path at top of editor.

Search/Replace & regex

  • Find in file vs Find in workspace.
  • Turn on Regex to do powerful matching.
  • Use include/exclude globs like: src/**, **/*.test.*, node_modules. 
Example regex ideas:
- Find TODO comments:         TODO:.*
- Find multiple spaces:       [ ]{2,}
- Find console logs (JS):     console\.log\([^)]*\)
- Match emails (simple):      [A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}
‚ö†Ô∏è Caution
Regex is powerful‚Äîtest carefully before ‚ÄúReplace All‚Äù across the workspace. Prefer using Source Control to review diffs.

Formatting & linting

  • Format Document runs a formatter (built-in or extension).
  • Format on Save automates formatting; best paired with a formatter like Prettier (web) or Black (Python).
  • Linting catches issues early (ESLint, Pylint/Ruff, etc.).
  • Use Problems view to navigate errors quickly.
‚ÑπÔ∏è Note
For team projects, commit formatter/linter config (e.g., .prettierrc, pyproject.toml, .editorconfig) and keep VS Code set to ‚Äúuse project tools‚Äù.

Files, terminals & tasks

Integrated terminal

  • Open terminal in the current workspace.
  • Create multiple terminals for different tasks (server, tests, git).
  • Split terminals to view outputs side-by-side.
  • Set default shell (PowerShell / bash / zsh) in settings.
üí° Tip
Use the terminal for running tests continuously while coding, and keep the Problems panel open for quick fixes.

Tasks

Tasks let you run build/test commands consistently from VS Code and bind them to shortcuts. 

// .vscode/tasks.json (example)
{
  "version": "2.0.0",
  "tasks": [
    {
      "label": "Run tests",
      "type": "shell",
      "command": "npm test",
      "group": "test",
      "problemMatcher": []
    }
  ]
}

Snippets

  • Snippets expand short triggers into boilerplate.
  • You can create language-specific snippets (e.g., JavaScript, Python).
  • Use tab stops like $1, $2 to jump through editable fields. 
// User snippet example (concept)
"Print to console": {
  "prefix": "clg",
  "body": ["console.log($1);"],
  "description": "Console log"
}

Git & GitHub workflows

Source Control view

  • Shows changed files, staged changes, and diffs.
  • Stage/unstage hunks and lines.
  • Write commit messages and commit from VS Code.
  • View history (often enhanced by extensions like GitLens).

Branches & conflicts

  • Switch branches from the Status Bar (bottom left).
  • Use the diff/merge editor to resolve conflicts.
  • After resolving conflicts: stage ‚Üí commit ‚Üí push.
‚ö†Ô∏è Caution
When resolving conflicts, make sure the final file compiles/runs and tests pass before pushing.
‚ÑπÔ∏è Note
If you use GitHub, VS Code integrates nicely with PRs/issues via the GitHub extensions (see Extensions section).

Debugging

How debugging works in VS Code

  • Create a launch configuration (Run & Debug ‚Üí create launch.json).
  • Set breakpoints and run your app in debug mode.
  • Use the debug toolbar to step through code.
  • Inspect variables, call stack, and watch expressions.

Common debug tools

  • Breakpoints (regular, conditional, logpoints)
  • Watch values change over time
  • Call stack to see how you got here
  • Debug Console to evaluate expressions

launch.json basics

VS Code uses .vscode/launch.json to store debug configurations. The exact setup depends on language/runtime. 

// Example idea (varies by language/extension)
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Launch Program",
      "type": "node",
      "request": "launch",
      "program": "${workspaceFolder}/index.js",
      "console": "integratedTerminal"
    }
  ]
}

Breakpoints & watch

  • Conditional breakpoint: break only when an expression is true.
  • Logpoint: print a message without stopping execution.
  • Watch: track complex expressions.
üí° Tip
Prefer logpoints for quick inspection, and conditional breakpoints for tricky loops or intermittent bugs.

Extensions

Suggested extensions (popular categories)

  • Language support: Python, Java, C/C++, Go, Rust, etc.
  • Web: Prettier, ESLint, HTML/CSS helpers.
  • Git: GitLens; GitHub Pull Requests & Issues.
  • Containers/Cloud: Docker, Kubernetes (if needed).
  • Productivity: Error Lens, TODO Tree, Path Intellisense.
‚ÑπÔ∏è Note
Keep extensions minimal in performance-sensitive environments; prefer project-local tooling for consistency.

Remote development

  • SSH: edit code on a remote server as if it‚Äôs local.
  • Dev Containers: open the repo inside a container with a consistent toolchain.
  • WSL: run Linux tools on Windows via WSL integration.

Common ‚ÄúI‚Äôm stuck‚Äù fixes

Formatting doesn‚Äôt work

  • Install a formatter extension (e.g., Prettier for web).
  • Set default formatter per-language.
  • Check bottom-right language mode matches your file.

Lint errors everywhere

  • Make sure you installed project dependencies.
  • Run the linter in terminal and compare.
  • Check ‚ÄúProblems‚Äù panel for exact messages.

Git not showing changes

  • Confirm you opened the correct folder (workspace root).
  • Check file is not excluded/ignored.
  • Verify git is installed and available in PATH.

Debugger won‚Äôt start

  • Confirm you installed the language debug extension.
  • Create/select the right launch configuration.
  • Try running the program in terminal first.
